oc-keysafe

给 opencode 的本地 Key 小金库 · Windows 单用户 · DPAPI 加密 · 零云同步

它解决什么问题

opencode 自定义 provider 的 API Key 默认明文存在 auth.json，{env:...} 替换在自定义 provider 上有 bug，{file:...} 又只能读明文。oc-keysafe 把所有 key 加密落盘，仅在 opencode 跑起来的那段时间释放成明文，opencode 一退出，明文立刻清空。

📋

🔌

☁️

工作流

STEP 01
init
建目录 + 设 ACL
STEP 02
add
输入 key（屏幕不回显）→ DPAPI 加密落盘
STEP 03
install-config
改 opencode.json 用 {file:...} 引用 runtime 路径
STEP 04
run
解密到 runtime → 启动 opencode → 等待
STEP 05
cleanup
opencode 退出 → 明文销毁 → 锁释放

opencode.json 集成示例

{
  "$schema": "https://opencode.ai/config.json",
  "provider": {
    "kiro": {
      "npm": "@ai-sdk/openai-compatible",
      "options": {
        "baseURL": "https://codecdn.bytecatcode.org/v1",
        "apiKey": "{file:C:/Users/<USER>/AppData/Local/oc-keysafe/runtime/bytecat-kiro.key}"
      },
      "models": { "claude-sonnet-4-5-20250929": {} }
    }
  }
}

路径用正斜杠避免 JSON 转义；oc-keysafe install-config 会自动算好这个路径。

5 分钟上手

# 1. 初始化
oc-keysafe init

# 2. 加 3 把 ByteCat 分组 key
oc-keysafe add bytecat-kiro   --note "Claude 系"
oc-keysafe add bytecat-codex  --note "GPT 系"
oc-keysafe add bytecat-gemini --note "Gemini 系"

# 3. 让 opencode.json 引用
oc-keysafe install-config --provider kiro   --key bytecat-kiro
oc-keysafe install-config --provider codex  --key bytecat-codex
oc-keysafe install-config --provider gemini --key bytecat-gemini

# 4. 体检
oc-keysafe check

# 5. 启动 opencode（以后只用这一条）
oc-keysafe run

FAQ

为什么不用主密码 / age / GPG？
主密码会引入"每次启动都要输入"的摩擦，违背"无感安全"目标；age/GPG 需要额外二进制依赖。DPAPI 是 Windows 内置的最小可行加密，等价于 Credential Manager 的底层。
密文复制到另一台电脑能用吗？
不能。DPAPI 主密钥绑定 Windows 账户与机器，迁移需要重新 add（这是预期行为）。
opencode 跑着的时候明文存在哪？
%LOCALAPPDATA%\oc-keysafe\runtime\，仅当前用户可读。opencode 退出立即清空。
如果 oc-keysafe run 被强杀（taskkill /F）？
注册了 PowerShell.Exiting 事件做兜底；极端情况残留时下次 run 会先清空。
能不能直接 opencode 启动而不走 run？
能启动，但 runtime 目录是空的，{file:...} 解析失败 → provider 无 key。必须走 oc-keysafe run。可在 PowerShell 配置文件加别名 opencode = oc-keysafe run。
支持 macOS / Linux 吗？
v1 不支持（DPAPI 是 Windows 专属）。后续版本会用 Keychain / Secret Service 补齐。
为什么不做 secure-erase？
SSD 的 wear-leveling 让任何用户态 secure-erase 都不可靠，做了反而误导。明文驻留时长已是最低。
